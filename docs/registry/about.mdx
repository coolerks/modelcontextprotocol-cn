<Note>
  MCP 注册表当前处于预览阶段。在正式发布之前可能发生破坏性变更或数据重置。如果您遇到任何问题，请在 [GitHub](https://github.com/modelcontextprotocol/registry/issues) 上报告。
</Note>

MCP 注册表是公开可访问的 MCP 服务器的官方集中式元数据仓库，由 MCP 生态系统的主要可信贡献者（如 Anthropic、GitHub、PulseMCP 和 Microsoft）共同支持。

MCP 注册表提供：

- 服务器创建者发布其服务器元数据的单一场所
- 通过 DNS 验证的命名空间管理
- 供 MCP 客户端和聚合器发现可用服务器的 REST API
- 标准化的安装和配置信息

服务器元数据以标准化的 [`server.json` 格式](https://github.com/modelcontextprotocol/registry/blob/main/docs/reference/server-json/draft/server.schema.json) 存储，包含：

- 服务器的唯一名称（例如 `io.github.user/server-name`）
- 服务器的位置（例如 npm 包名称、远程服务器 URL）
- 执行指令（例如命令行参数、环境变量）
- 其他发现数据（例如描述、服务器能力）

![MCP 注册表生态系统](./ecosystem-diagram.excalidraw.svg)

### 与包注册表的关系

包注册表——如 npm、PyPI 和 Docker Hub——托管包含代码和二进制文件的包。

MCP 注册表托管指向这些包的元数据。

例如，一个 `weather-mcp` 包可以托管在 npm 上，MCP 注册表中的元数据可以将"weather v1.2.0"服务器映射到 `npm:weather-mcp`。

[包类型指南](./package-types) 列出了支持的包类型和注册表。未来可能会根据社区需求支持更多的包注册表。如果您有兴趣为某个包注册表构建支持，请[创建一个 issue](https://github.com/modelcontextprotocol/registry)。

### 与服务器开发者的关系

MCP 注册表同时支持开源和闭源服务器。服务器开发者可以将其服务器的元数据发布到注册表，只要服务器的安装方法是公开可用的（例如 npm 包或公共注册表上的 Docker 镜像）_或_ 服务器本身是公开可访问的（例如不限于私有网络的远程服务器）。

MCP 注册表**不支持**私有服务器。私有服务器是那些只有少数用户可以访问的服务器。例如，发布在私有网络上（如 `mcp.acme-corp.internal`）或私有包注册表上（如 `npx -y @acme/mcp --registry https://artifactory.acme-corp.internal/npm`）的服务器。如果您想发布私有服务器，我们建议您托管自己的私有 MCP 注册表并将它们添加到那里。

### 与下游聚合器的关系

MCP 注册表旨在主要被下游聚合器（如 MCP 服务器市场）消费。

MCP 注册表托管的元数据故意保持非主观性。下游聚合器可以提供策展或额外的元数据，如社区评分。

我们期望下游聚合器会定期但不频繁地（例如每小时一次）使用 MCP 注册表 API 拉取新的元数据。更多信息请参阅 [MCP 注册表聚合器指南](./registry-aggregators)。

### 与其他 MCP 注册表的关系

除了公共 REST API 之外，MCP 注册表还定义了一个 [OpenAPI 规范](https://github.com/modelcontextprotocol/registry/blob/main/docs/reference/api/openapi.yaml)，其他 MCP 注册表可以实现该规范以为 MCP 主机应用程序提供标准化接口。

我们期望许多下游聚合器会实现此接口。私有 MCP 注册表也可以实现它以受益于现有主机应用程序的支持。

请注意，官方 MCP 注册表代码库**不是**为自托管设计的，注册表维护者无法为此用例提供支持。如果您选择 fork 它，您需要独立维护和运营。

### 与 MCP 主机应用程序的关系

MCP 注册表不旨在被主机应用程序直接消费。相反，主机应用程序应通过符合官方 MCP 注册表 OpenAPI 规范的 REST API 消费其他 MCP 注册表，如下游市场。

## 信任与安全

### 验证服务器真实性

MCP 注册表使用命名空间身份验证来确保服务器来自其声称的来源。服务器名称遵循反向 DNS 格式（如 `io.github.username/server` 或 `com.example/server`），将它们关联到已验证的 GitHub 账户或域名。

此命名空间系统确保只有 GitHub 账户或域名的合法所有者才能在该命名空间下发布服务器，为生态系统提供信任和问责制。有关身份验证方法的详细信息，请参阅[身份验证指南](./authentication)。

### 安全扫描

MCP 注册表将安全扫描委托给：

- **底层包注册表**——npm、PyPI、Docker Hub 和其他包注册表执行其自身的安全扫描和漏洞检测。
- **下游聚合器**——MCP 注册表聚合器和市场可以实现额外的安全检查、评级或策展。

MCP 注册表专注于命名空间身份验证和元数据托管，同时依赖更广泛的生态系统对实际服务器代码进行安全扫描。

### 垃圾信息预防

MCP 注册表使用多种机制来防止垃圾信息：

- **命名空间身份验证要求**——发布者必须通过 GitHub、DNS 或 HTTP 验证其命名空间的所有权，防止任意的垃圾信息提交。
- **字符限制和验证**——自由格式字段有严格的字符限制和正则表达式验证以防止滥用。
- **手动下架**——注册表维护者可以手动删除垃圾或恶意服务器。有关删除内容的详细信息，请参阅[审核策略](./moderation-policy)。

未来考虑的垃圾信息预防措施包括更严格的速率限制、基于 AI 的垃圾信息检测和社区举报功能。
