## SEP 优先级

**属于以下优先领域的 SEP 将获得加速审查，并且最有可能被接受。** 不属于这些领域的 SEP 不会自动被拒绝，但贡献者应预期更长的审查时间和更高的论证标准。维护者的能力是有限的。我们首先将资源投入到这些优先级上。

如果您正在考虑提交 SEP，请检查它是否与以下领域之一相符，在相关的[工作组或兴趣组](/community/working-interest-groups)中讨论，并获得该组的支持。有工作组支持且与路线图有明确关联的 SEP 进展最快。请参阅 [SEP 指南](/community/sep-guidelines)了解完整流程。

## 优先领域

### 1. 传输演进与可扩展性

Streamable HTTP 为 MCP 提供了生产就绪的传输，但在规模运行时暴露了水平扩展、无状态操作和中间件模式方面的不足。

**我们想要实现的目标：**

- **下一代传输**：演进 Streamable HTTP 以跨多个服务器实例无状态运行，并在负载均衡器和代理后面正确运行。
- **可扩展的会话处理**：定义如何创建、恢复和迁移会话，使服务器重启和扩展事件对连接的客户端透明。
- **MCP 服务器卡片**：一种通过 `.well-known` URL 公开结构化服务器元数据的标准，使浏览器、爬虫和注册表可以在不连接的情况下发现服务器的能力。

**工作组归属：**

- **传输工作组**负责传输和会话工作：一系列涵盖线路格式、会话模型和恢复协议的 SEP，以及针对 SDK 作者的合规性指南。
- **服务器卡片工作组**负责服务器卡片格式及其分发，与更广泛的行业 AI 目录工作协调。

我们**不会**在本周期引入额外的官方传输。保持较小的集合可保护生态系统的兼容性；社区应通过自定义传输进行实验。

### 2. 代理通信

任务原语（[SEP-1686](https://github.com/modelcontextprotocol/modelcontextprotocol/issues/1686)）为代理提供了可靠的即时调用/后续获取模式。在生产中运行暴露了生命周期语义方面的不足，**代理工作组**应弥补这些不足：

- **重试语义**：当任务暂时失败时会发生什么，以及由谁决定是否重试。
- **过期策略**：完成后结果保留多长时间，以及客户端如何得知结果已过期。

这些是我们今天可以指出的不足。代理工作组还应收集和分类生产部署中的操作问题——随着生态系统中更多的规模化任务运行，此列表将会增长。

### 3. 治理成熟化

MCP 已发展成为 Linux 基金会下的多公司开放标准。[SEP-1302](https://github.com/modelcontextprotocol/modelcontextprotocol/issues/1302) 正式确立了工作组和兴趣组，[SEP-2085](https://github.com/modelcontextprotocol/modelcontextprotocol/issues/2085) 建立了继任和修正程序。下一步是为社区提供清晰的领导路径，使项目不依赖于少数个人。

**治理工作组**应交付：

- **贡献者阶梯 SEP**，定义从社区参与者 → 工作组贡献者 → 工作组促进者 → 主导维护者 → 核心维护者的晋升路径，每一步都有明确的提名和评审标准。
- **委托模型**，允许具有良好记录的工作组在其领域内接受 SEP 和发布扩展更新，而无需完整的核心维护者审查周期。
- **章程模板**，每个工作组和兴趣组公开维护：范围、活跃交付物、成功标准和退休条件，每季度审查一次。

### 4. 企业就绪

企业正在大规模部署 MCP，并遇到了协议尚未解决的不足。

我们需要明确问题陈述和方向性提案的领域：

- **审计追踪和可观测性**：对客户端请求了什么以及服务器做了什么提供端到端的可见性，以企业可以输入其现有日志和合规性管道的形式。
- **企业托管授权**：从静态客户端密钥到 SSO 集成流程（[Cross-App Access](https://xaa.dev)）的铺平路径，使 IT 部门能够像管理其他一切一样管理 MCP 访问。
- **网关和代理模式**：当客户端不直接连接到服务器而是通过中间人路由时的明确行为。这可能包括授权传播、会话语义以及网关允许查看的内容。
- **配置可移植性**：一种一次性配置服务器并使该配置在不同 MCP 客户端间工作的方法。

我们预计将成立一个**企业工作组**来负责此领域。大部分产出很可能以扩展的形式落地，而非核心规范的变更。

## 展望

这些领域有社区和核心维护者的兴趣，但不是最高优先级。我们将支持社区在其中任何一个领域成立工作组，并在时间允许的情况下审查这些主题的 SEP。

- **触发器和事件驱动更新**——客户端目前通过轮询或保持 SSE 连接打开来了解服务器端状态变化。标准化的回调机制（webhook 或类似）将让服务器在新数据可用时主动通知客户端，并在所有传输上具有定义的排序保证。
- **结果类型改进**——工具调用、资源读取和任务结果都以完整内联的方式到达。流式结果将让客户端在交互场景（生成的文本、音频、视频帧）中增量接收输出；基于引用的结果将让客户端决定何时将大型有效负载拉取到上下文中，而不是默认污染上下文。这是跨领域的：流式涉及传输，引用涉及模式。
- **安全与授权**——更细粒度的最小权限范围、关于避免 OAuth 混淆攻击的更清晰指南、客户端和服务器上的安全凭据管理，以及通过 Linux 基金会路由的社区驱动的漏洞披露计划。赞助工作已经在进行中：[SEP-1932 (DPoP)](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/1932) 和 [SEP-1933 (工作负载身份联合)](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/1933)。
- **扩展生态系统**——`ext-auth` 和 `ext-apps` 轨道是扩展机制有效性的早期证明。使其成熟、研究用于组合能力的技能原语、以及向注册表添加一流的扩展支持，都将加强从实验到标准的路径。

## 验证

协议规范的好坏取决于遵循它的实现。除了上述领域外，我们将继续投资于：

- **合规性测试套件**：自动验证客户端、服务器和 SDK 是否正确实现规范，覆盖范围随着每个新功能领域而扩展。
- **SDK 层级**：[SEP-1730](https://github.com/modelcontextprotocol/modelcontextprotocol/issues/1730) 引入的分层系统为开发者提供了哪些 SDK 最紧密跟踪规范的明确信号。
- **参考实现**：新功能的规范实现，以锚定社区发展并解除早期采用者的阻碍。

- **加入工作组或兴趣组**：查看[工作组和兴趣组](/community/working-interest-groups)页面和[社区沟通渠道](/community/communication)，与上述每个领域活跃的组建立联系。

- **为项目做贡献**：阅读[贡献指南](/community/contributing)了解如何参与规范、SDK 和工具的开发。
