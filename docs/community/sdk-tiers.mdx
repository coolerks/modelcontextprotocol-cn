MCP SDK 分层系统为官方和社区驱动的 SDK 建立了功能完整性、协议支持和维护承诺的明确期望。
这有助于开发者选择适合其需求的 SDK，并为 SDK 维护者提供改善采用期望的清晰路径。

<Note>
  **关键日期：**
  - **2026 年 1 月 23 日**：一致性测试可用
  - **2026 年 2 月 23 日**：官方 SDK 分层发布

在 1 月 23 日至 2 月 23 日期间，SDK 维护者可以与
一致性测试工作组合作，采用测试并使用下面定义的标准化标签设置 GitHub 问题跟踪。

</Note>

## 概述

SDK 根据功能完整性、维护承诺和文档质量分为三个层级：

- **第 1 层**：完全支持的 SDK，具有完整的协议实现，包括所有非实验性功能
  和可选能力，如采样和引导
- **第 2 层**：积极维护的 SDK，致力于完全支持协议规范
- **第 3 层**：实验性、部分实现或专用 SDK

实验性功能（如 Tasks）和协议扩展（如 MCP Apps）对任何层级都不是必需的。

## 层级要求

| 要求              | 第 1 层：完全支持                                  | 第 2 层：致力于完全支持                      | 第 3 层：实验性 |
| ----------------- | -------------------------------------------------- | -------------------------------------------- | --------------- |
| **一致性测试**    | 100% 通过率                                        | 80% 通过率                                   | 无最低要求      |
| **新协议功能**    | 在新规范版本发布前，根据功能复杂度按发布约定时间线 | 6 个月内                                     | 无时间线承诺    |
| **问题分类**      | 2 个工作日内                                       | 一个月内                                     | 无要求          |
| **关键 Bug 解决** | 7 天内                                             | 两周内                                       | 无要求          |
| **稳定版本**      | 需要，具有清晰的版本控制                           | 至少一个稳定版本                             | 不需要          |
| **文档**          | 全面，包含所有功能的示例                           | 涵盖核心功能的基础文档                       | 无最低要求      |
| **依赖策略**      | 已发布的更新策略                                   | 已发布的更新策略                             | 不需要          |
| **路线图**        | 已发布的路线图                                     | 已发布的达到第 1 层的计划或仍为第 2 层的说明 | 不需要          |

**问题分类**意味着标记并确定问题是否有效，而非解决问题。

**关键 Bug**指 P0 问题（详细标准请参见[优先级标签](#仅对可操作的问题使用)）。

**稳定版本**是明确标记为生产就绪的已发布版本（例如，版本 `1.0.0`
或更高，没有 `-alpha`、`-beta` 或 `-rc` 等预发布标识符）。

**清晰的版本控制**意味着遵循惯用的版本控制模式，并附有文档化的
破坏性更改策略，以便用户在升级时了解兼容性预期。

**路线图**概述了跟踪所需 MCP 规范组件（非实验性功能和可选能力，如
[一致性测试](#一致性测试)中所述）实施的具体步骤和工作项，使用户能够了解即将到来的功能支持。

## 一致性测试

所有 SDK 都使用[自动化一致性测试](https://github.com/modelcontextprotocol/conformance)进行评估，
这些测试根据已发布的规范验证协议支持。SDK 根据测试结果获得一致性评分：

- **第 1 层**：需要 100% 一致性
- **第 2 层**：需要 80% 一致性
- **第 3 层**：无最低要求

一致性评分仅针对**适用的必需测试**计算：

- SDK 所针对规范版本的测试
- 排除标记为待定或跳过的测试
- 排除实验性功能的测试
- 排除遗留向后兼容性测试（除非 SDK 声明遗留支持）
- 在争议解决之前，排除一致性仓库中标记为 `disputed` 的测试

一致性测试通过运行标准化测试场景并检查协议消息交换来验证 SDK 是否正确实现了协议。
有关如何处理临时测试失败，请参阅[层级降级](#层级降级)。

## 层级晋升

SDK 维护者可以通过以下方式请求层级晋升：

1. 根据层级要求进行自我评估
2. 在 [modelcontextprotocol/modelcontextprotocol](https://github.com/modelcontextprotocol/modelcontextprotocol) 仓库中开设带有支持证据的议题
3. 通过自动化一致性测试
4. 获得 SDK 工作组维护者的批准

SDK 工作组审查晋升请求并做出最终的层级分配。

## 层级降级

如果最新稳定版本上的现有一致性测试连续失败 4 周，SDK 可能被降级到较低的层级：

- **第 1 层 → 第 2 层**：任何一致性测试失败
- **第 2 层 → 第 3 层**：超过 20% 的一致性测试失败

如果问题在两个月内仍未解决，SDK 也可能被降级。

## 问题分类标签

SDK 仓库必须使用一致的标签，以实现问题处理指标的自动化报告。
层级计算使用这些指标来衡量分类响应时间（从问题创建到第一个标签的时间）和关键 Bug 解决时间（从 P0 标签到问题关闭的时间）。

### 类型（选择一个）

| 标签          | 描述                 |
| ------------- | -------------------- |
| `bug`         | 某些功能无法正常工作 |
| `enhancement` | 请求新功能           |
| `question`    | 需要更多信息         |

使用 [GitHub 原生问题类型](https://docs.github.com/en/issues/tracking-your-work-with-issues/using-issues/managing-issue-types-in-an-organization)的仓库无需类型标签即可满足此要求。

### 状态（选择一个）

在所有仓库中使用这些确切的标签名称，以实现一致的报告和分析。

| 标签                 | 描述                   |
| -------------------- | ---------------------- |
| `needs confirmation` | 不确定是否仍然相关     |
| `needs repro`        | 信息不足以复现         |
| `ready for work`     | 有足够的信息可以开始   |
| `good first issue`   | 适合新手               |
| `help wanted`        | 欢迎熟悉代码库的人贡献 |

### 优先级（仅对可操作的问题使用）

| 标签 | 描述                                 |
| ---- | ------------------------------------ |
| `P0` | 关键：核心功能故障或高严重性安全漏洞 |
| `P1` | 影响许多用户的重大 Bug               |
| `P2` | 中等问题，有价值的功能请求           |
| `P3` | 锦上添花，罕见的边缘情况             |

**P0（关键）** 问题是：

- **安全漏洞**，CVSS 评分 ≥ 7.0（高或严重）
- **核心功能故障**，阻止基本的 MCP 操作：连接建立、消息交换或使用核心原语（工具、资源、提示）
