Model Context Protocol (MCP) 遵循正式的治理模型，以确保透明的决策和社区参与。本文档概述了项目的组织方式以及决策的制定方式。

## 总体项目政策

Model Context Protocol 已成立为 **LF Projects, LLC 的一个系列**。适用于 Model Context Protocol 及其参与者的政策，包括商标使用指南，位于 [https://www.lfprojects.org/policies/](https://www.lfprojects.org/policies/)。根据本治理文件的规定批准的治理变更也必须得到 LF Projects, LLC 的批准。

Model Context Protocol 参与者确认，所有新贡献的版权将由版权持有人作为独立著作物保留，并且不会要求任何贡献者或版权持有人将版权转让给项目。

除非下文另有说明，否则对项目的所有代码和规范贡献必须使用 Apache 许可证 2.0 版（此处提供：[https://www.apache.org/licenses/LICENSE-2.0](https://www.apache.org/licenses/LICENSE-2.0)）（"项目许可证"）。

所有对外发布的代码和规范将在项目许可证下提供。核心维护者可以例外批准对内向或外向贡献使用替代开源许可证。

所有文档（不包括规范）将在 Creative Commons Attribution 4.0 International 许可证下提供，网址为：https://creativecommons.org/licenses/by/4.0。

## 技术治理

MCP 项目采用层级结构，类似于 Python、PyTorch 和其他开源项目：

| 角色                  | 范围              |
| --------------------- | ----------------- |
| **主导维护者 (BDFL)** | 最终决策权        |
| **核心维护者**        | 项目总体方向      |
| **维护者**            | 工作组、SDK、组件 |
| **贡献者**            | 议题、PR、讨论    |

- **贡献者**提交议题、创建拉取请求，并为项目做出贡献。
- **维护者**推动 MCP 项目内的组件，如 SDK、文档和工作组。
- **核心维护者**推动项目的总体方向，并监督贡献者和维护者。
- **主导维护者**是最终决策者（也称为 BDFL - 终身仁慈独裁者）。

维护者、核心维护者和主导维护者共同组成 **MCP 指导组**。

所有维护者都应强烈倾向 MCP 的设计理念。技术治理过程中的成员资格是针对个人而非公司的。也就是说，没有为特定公司保留席位，成员资格与个人相关联，而非雇用该人的公司。

### 沟通渠道

技术治理通过所有维护者共享的 [Discord 服务器](https://discord.gg/6CSzBmMkjX) 进行协调。每个维护者组可以选择额外的沟通渠道，但所有决策及其支持讨论必须记录并在 Discord 服务器上透明地提供。

### 角色

[贡献者阶梯](/community/contributor-ladder)是每个角色的规范定义——其要求、职责、特权、晋升流程和不活跃政策。本节从概念上概述了这些角色与治理的关系。

**维护者**管理特定领域，例如 SDK、文档或[工作组](/community/working-interest-groups)。他们独立地为其领域做出决策，并在需要时升级到核心维护者。维护者对其各自的仓库具有写入权限。

**核心维护者**指导 MCP 规范和项目的总体方向。他们可以通过多数投票否决维护者的决策、解决争议以及任命或移除维护者。核心维护者对所有 MCP 仓库具有管理权限，但使用与外部贡献者相同的拉取请求工作流程。

**主导维护者**拥有最终权力，可以否决核心维护者或维护者的任何决策——这通常被称为终身仁慈独裁者 (BDFL)。主导维护者任命和移除核心维护者，并且是所有项目基础设施的管理员。他们是核心维护者组的一部分，并应公开阐明其推理。

[贡献者阶梯](/community/contributor-ladder)还定义了**成员**和**社区版主**角色，这些角色不属于指导组。

### 决策流程

核心维护组每两周举行一次会议，讨论和投票提案，并讨论任何必要的主题。如果需要，共享的 Discord 服务器可用于讨论和投票较小的提案。

主导维护者、核心维护者和维护者组应每三到六个月尝试举行一次面对面会议。

## 流程

核心维护者和主导维护者负责 Model Context Protocol 的所有方面，包括文档、议题、内容建议以及 [MCP 项目](https://github.com/modelcontextprotocol) 下的所有其他部分。维护者负责其 MCP 项目领域的文档、议题和内容建议，但鼓励参与 MCP 项目的一般维护。

维护者、核心维护者和主导维护者应使用与外部贡献者相同的贡献流程，而不是直接对仓库进行更改。这提供了对意图的洞察和讨论的机会。

### 工作组和兴趣小组

MCP 的协作和贡献围绕两种结构组织：[工作组和兴趣小组](/community/working-interest-groups)。

- **兴趣小组**通过公开讨论识别和阐明 MCP 应解决的问题
- **工作组**通过产生 SEP 或实现等交付物来开发具体解决方案

有关如何创建、参与和协调这些小组的详细信息，请参阅[工作组和兴趣小组](/community/working-interest-groups)文档。

### 规范增强提案 (SEP)

对规范的拟议更改必须以[规范增强提案 (SEP)](/community/sep-guidelines) 的形式提交。SEP 是提出主要新功能、收集社区意见和记录设计决策的主要机制。

有关完整的 SEP 流程、格式要求和状态工作流程，请参阅 [SEP 指南](/community/sep-guidelines)。

### 维护职责

没有专门维护者的组件（如文档）属于核心维护者的责任。这些组件通过拉取请求遵循标准的贡献指南，由维护者处理评审，并将任何重大更改升级到核心维护者评审。

鼓励核心维护者和维护者改进 MCP 项目的任何部分，无论正式的维护分配如何。

## 沟通

### 核心维护者会议

核心维护组每两周举行一次会议，讨论提案和项目。提案的说明应公开。核心维护组将努力每 3-6 个月举行一次面对面会议。

### 公共聊天

MCP 项目维护着一个[公共 Discord 服务器](https://discord.gg/6CSzBmMkjX)，为兴趣小组提供开放的聊天频道。MCP 项目可能为某些通信设有私密频道。

## 提名、确认和移除维护者

维护者组的成员资格是基于功绩授予**个人**的，在证明了专业知识并符合 MCP 方向之后。成员资格与个人而非其雇主相关联，并且没有任期限制。

每个角色的提名流程、赞助要求、评审时间表和不活跃标准在[贡献者阶梯的晋升流程](/community/contributor-ladder#advancement-process)中定义。
