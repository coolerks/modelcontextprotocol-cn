在 MCP 贡献者社区中，我们维护两种协作形式：**兴趣小组 (IG)** 和 **工作组 (WG)**。

## 快速参考

|              | 兴趣小组 (IG)            | 工作组 (WG)                          |
| ------------ | ------------------------ | ------------------------------------ |
| **目的**     | 识别和讨论问题           | 构建具体解决方案                     |
| **产出**     | 问题陈述、用例、建议     | SEP、实现、代码                      |
| **投入度**   | 期望积极参与             | 期望积极参与                         |
| **持续时间** | 只要主题相关就持续存在   | 直到交付物完成                       |
| **领导层**   | 协调人                   | 负责人                               |
| **决策方式** | 粗略共识，非约束性       | 具有约束力（默许共识 → 投票 → 升级） |
| **示例**     | "MCP 安全"——讨论安全挑战 | "服务器身份"——实现身份验证           |

## 何时使用哪种形式

**加入兴趣小组，当你：**

- 遇到问题但不确定解决方案
- 想探索某个想法是否得到社区支持
- 是 MCP 新手，想了解某个主题领域
- 想分享用例和需求

**加入工作组，当你：**

- 有具体的解决方案要实现
- 准备好编写代码或 SEP
- 能够定期投入时间进行积极开发
- 想帮助构建特定功能

**典型流程**：在 IG 中讨论问题 → 验证问题值得解决 → 组建或加入 WG 构建解决方案 → 提交 SEP → 实施

## 兴趣小组 (IG)

**目标：** 促进在特定主题上分享兴趣的 MCP 贡献者之间的讨论和知识共享。重点是识别值得解决的问题和收集需求——而非构建解决方案。

**IG 的工作：**

- 在 Discord 频道中主持讨论
- 定期举行会议分享用例
- 记录问题陈述和需求
- 就优先事项达成共识
- 为工作组和 SEP 提供输入

**示例：**

- MCP 安全
- MCP 身份验证
- 在企业环境中使用 MCP
- 托管 MCP 客户端的工具和实践

## 工作组 (WG)

**目标：** 在 SEP、一系列相关 SEP 或官方认可的项目上进行协作。WG 产生具体的交付物。

**WG 的工作：**

- 编写和迭代 SEP
- 构建参考实现
- 维护持续进行的项目（Inspector、Registry、SDK）
- 推动功能从提案到规范

**示例：**

- Registry（注册中心）
- Inspector（检查器）
- Tool Filtering（工具过滤）
- Server Identity（服务器身份）

## 治理

以下规则适用于所有 MCP 工作组和兴趣小组。各个小组的章程不能覆盖这些要求。如果 WG 和 IG 的规则有所不同，会明确注明。

### 领导层

每个小组有一名或多名**负责人**（兴趣小组中称为**协调人**）。

**对所有负责人和协调人的要求：**

- 至少拥有 MCP 贡献者阶梯中的成员身份——角色定义请参阅[治理](/community/governance)
- 证明对该小组范围内领域的持续参与
- 能够跨组织边界进行协调
- 致力于运营该小组
- 小组及其领导层由至少两名核心维护者或一名主导维护者赞助

**对 WG 负责人的额外要求：**

- 每周投入 2-3 小时用于 WG 活动

**所有负责人的职责：**

- 安排并协调定期会议
- 与参与者协作制定议程并提前发布
- 确保会议记录在 48 小时内发布
- 维护小组的文档
- 在[访问仓库](https://github.com/modelcontextprotocol/access)中维护成员列表和相应的访问列表
- 主动招募并保留跨组织和视角的广泛代表性成员

**WG 负责人的额外职责：**

- 推动提案通过[SEP 流程](/community/sep-guidelines)直至解决
- 对 WG 范围内的 SEP 进行分类，包括关闭不符合路线图的 SEP（附有记录的理由；作者可以向核心维护者上诉）
- 将受阻的决策以清晰的上下文升级到核心维护者
- 维护工作组的路线图
- 持续向一位或多位核心维护者征求小组总体方向的反馈
- 向社区和核心维护组提供季度状态更新

### 参与级别

所有小组使用以下参与层级。请注意，**WG 成员**是特定于小组的参与级别，不同于组织范围的**成员**角色——个人可以在特定小组中担任 WG 成员而无需拥有组织范围的成员身份，反之亦然。

| 级别              | 描述                           | 权限                               |
| ----------------- | ------------------------------ | ---------------------------------- |
| **观察者**        | 任何有兴趣关注小组工作的人     | 读取权限，可参加会议，讨论参与有限 |
| **参与者**        | 小组讨论的积极贡献者           | 可提议议程项目，参与异步投票       |
| **WG 成员**       | 持续贡献者，具有证明的专业知识 | 计入法定人数（仅限 WG）            |
| **负责人/协调人** | 小组的运营领导                 | 制定议程、协调、升级               |

兴趣小组主要与观察者、参与者和协调人一起运作。如果 IG 的工作需要正式决策，可以采用 WG 成员层级，但并非必需。

**成为 WG 成员**（WG 以及采用 WG 成员层级的 IG）：

- 持续参与超过 3 个月
- 有意义的贡献（代码、规范文本、评审或文档）
- 由现有 WG 成员或负责人提名
- 负责人在 7 天内没有异议

**WG 成员职责：**

- 继续诚信贡献
- 在各小组的成员列表中维护姓名、组织和 Discord 名称

**活跃与荣誉状态：** 连续 3 个月未参与的 WG 成员将被移至荣誉状态，可通过重新展示参与来回归。

### 决策流程

本节主要适用于做出具有约束力决策（对技术设计、规范变更等的共识）的工作组。兴趣小组通常通过讨论中的粗略共识运作，不做出具有约束力的决策——其产出是建议、问题陈述和用例。采用 WG 成员层级的 IG 可以在内部决策中使用此流程。

**WG 共识**通过以下递进方式实现。在进入下一步之前，会先尝试当前步骤。

<Steps>
  <Step title="默许共识（默认）">
    - 提案在明确的截止日期前公布（小事项至少 5 天，重大事项 10 天）
    - 沉默即表示同意
    - 任何 WG 成员可以提出有记录的反对意见
    - 反对意见必须提出替代方案或明确的解决方案标准
    - 如果截止日期前没有反对意见，则提案被接受
  </Step>
  <Step title="正式投票（当默许共识受阻时）">
    当 WG 成员在默许共识期间提出反对，或者负责人或三名及以上 WG 成员要求时，将触发正式投票。

    - 法定人数：50% 的活跃 WG 成员
    - 通过条件：常规事项简单多数；范围变更需 2/3 多数
    - 核心维护者的反馈是建议性的，除非明确指出具有约束力
    - 所有投票均有理由记录

  </Step>
  <Step title="升级（当投票无法解决时）">
    如果投票未能解决问题（没有法定人数、未通过或结果有争议），负责人将按照下面的升级路径升级到核心维护者。
  </Step>
</Steps>

### 升级路径

对于小组范围内的技术和设计分歧，小组应在涉及核心维护者之前先在本地解决。对于 WG，这意味着使用上述决策流程。对于 IG，协调人应在升级之前尝试达成粗略共识。

某些分歧不适合在小组层面解决，应直接升级到核心维护者：

- 范围争议（某个主题是否属于小组章程范围）
- 权限争议（小组是否有权决定某个事项）
- 跨组冲突（涉及多个 WG 或 IG 的分歧）
- 行为准则或行为问题
- 成员资格或参与争议

当需要升级时：

1. 负责人记录决策、考虑的选项和分歧点
2. 负责人以明确的请求向核心维护组提交升级
3. 核心维护组指定一名核心维护者——不应与相关方有组织关联——来解决该问题并向小组报告
4. 被指定的核心维护者：（a）提供具有约束力的指导，（b）要求提供更多信息，或（c）建议整个核心维护组进行审议
5. 时间线：升级应在 5 个工作日内收到初步回复

### 会议要求

负责人根据小组的当前需求和生命周期阶段确定会议频率、形式和时长。没有固定的节奏要求——接近规范发布的工作组可能每周开会，而处于早期探索阶段的 IG 可能每月开会或主要异步工作。

无论形式或频率如何，所有小组会议必须：

- 对所有社区参与者开放（不得是闭门或组织内部会议）
- 至少在 [meet.modelcontextprotocol.io](https://meet.modelcontextprotocol.io) 上提前 7 天发布
- 议程公开可用。议程或议程链接应作为[会议笔记类别中的 GitHub Discussion](https://github.com/modelcontextprotocol/modelcontextprotocol/discussions/) 发布
- 会议记录在 48 小时内发布到同一讨论中

负责人应积极让 WG 成员和参与者参与运营职责，如准备议程、记录会议纪要和协调讨论。

### 沟通渠道

所有小组使用以下渠道：

| 渠道                                 | 用途           | 回复期望 |
| ------------------------------------ | -------------- | -------- |
| Discord `#{name}-wg` 或 `#{name}-ig` | 快速提问、协调 | 尽力而为 |
| GitHub Discussions                   | 长篇技术讨论   | 每周分类 |

除 Discord 外，小组可以在 [GitHub Discussions](https://github.com/modelcontextprotocol/modelcontextprotocol/discussions/) 中建立讨论类别。负责人将被授予相应的角色来管理和主持讨论。

### 报告

**工作组**提供季度更新（1 月、4 月、7 月、10 月底），包括：

- 交付物进度
- 受阻项和升级情况
- 成员变更
- 即将到来的优先事项
- 资源需求

季度更新以文档形式发布在工作组的 [GitHub Discussions](https://github.com/modelcontextprotocol/modelcontextprotocol/discussions/) 类别中。可选地在核心维护者会议上与核心维护者讨论。

**兴趣小组**没有正式的报告要求，但应保持其章程和成员列表为最新状态。

### 生命周期

**工作组的组建：**

- 必须有一个广泛认可的需要协调的问题
- 创建 WG 到 `docs/community/<name>/overview.mdx` 的 PR，由 CODEOWNERS 控制，需要维护者批准
- 创建章程到 `docs/community/<name>/charter.mdx` 的 PR，由 CODEOWNERS 控制，需要核心维护者批准
- 初始成员列表由 WG 负责人批准

**兴趣小组的组建：**

- 在 [Discord](https://discord.gg/6CSzBmMkjX) 的 `#wg-ig-group-creation` 频道中填写创建模板
- 核心维护者审查提案；IG 及其协调人必须由至少两名核心维护者或一名主导维护者赞助
- 获得赞助后，协调人组织 IG 并创建章程

**解散：**

- **WG**：WG 负责人或核心维护者提出解散及理由；需要核心维护者或主导维护者批准。WG 在持续一段时间没有活跃工作或完成所有计划交付物时也会被解散。
- **IG**：核心维护者或主导维护者可以解散不再活跃或不再需要的 IG。
- 在这两种情况下，文档被归档，频道被标记为不活跃。

### 章程修订

对小组章程（WG 或 IG）的更改需要：

- 由负责人/协调人或核心维护者提议
- 核心维护者批准

## 章程

每个 MCP 工作组和兴趣小组必须维护一份章程文档，记录其具体使命、范围、领导层、成员和运营。上述治理规则自动适用，无需在章程中重复。

有关所需结构和可复制模板，请参阅[小组章程模板](/community/charter-template)。

## 常见问题

### 如何参与 MCP 贡献？

这些组提供了一个入口：

1. [加入 Discord](https://discord.gg/6CSzBmMkjX) 并关注与你相关的 IG。参加[实时会议](https://meet.modelcontextprotocol.io/)。参与讨论。
2. 主动协助运营职责——协调会议、准备议程、做笔记。在 SEP 讨论中分享你的用例。
3. 准备好动手工作时，为 WG 交付物做出贡献。
4. 持续贡献是获得 WG 成员身份和贡献者阶梯晋升的公认途径。

### 我在哪里可以找到所有当前 WG 和 IG 的列表？

在 [MCP 贡献者 Discord](https://discord.gg/6CSzBmMkjX) 上，有为每个工作组和兴趣小组划分的频道区域。已制定章程的小组在 [modelcontextprotocol 仓库](https://github.com/modelcontextprotocol/modelcontextprotocol)的 `docs/community/` 下也有文档。

### 在开始 WG 之前，我需要加入 IG 吗？

不需要。IG 参与可以帮助验证想法和建立支持，但这不是必需的。如果你有明确的交付物并能获得核心维护者的赞助，你可以直接提出 WG 提案。

### 我需要加入 WG 才能提交 SEP 吗？

不需要。任何人都可以提交 SEP。但是，WG 协作可以加强你的提案并帮助它找到赞助者。

### 如果我的 IG 讨论导致了具体的解决方案怎么办？

你可以：

- 组建一个新的 WG 来构建解决方案
- 如果现有 WG 涵盖该领域，则加入其中
- 如果解决方案定义明确，直接提交 SEP

### 一个人可以同时加入多个 IG/WG 吗？

可以。在你时间允许的情况下参与尽可能多的小组。
