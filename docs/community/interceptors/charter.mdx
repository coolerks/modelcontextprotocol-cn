## 小组类型

**工作组**

## 使命宣言

拦截器工作组旨在标准化上下文操作在代理生命周期关键点被拦截、验证和转换的方式。这涵盖 MCP 定义的操作，如工具调用、资源访问、提示处理、采样和引导，以及任何其他塑造代理上下文的操作——包括 LLM 完成和自定义应用特定工作流。生态系统正在开发大量用于横切关注点的边车、代理和网关，这些大多是不可复用和不可互操作的，造成了 M × N 集成问题。WG 将产生规范扩展和参考实现，将拦截器定义为一种新的 MCP 原语，具有两种类型——验证器（检查并返回通过/失败决定）和修改器（转换上下文载荷）——可通过 MCP 现有的 JSON-RPC 模式发现和调用，跨部署模型包括进程内、边车和远程服务。

## 范围

### 范围内

- **规范工作**：定义拦截器原语的 SEP——验证器和修改器类型、MCP 操作的生命周期事件钩子（工具调用、资源读取、提示获取、采样、引导）并可扩展到非 MCP 上下文操作（LLM 完成、自定义工作流）、信任边界感知执行模型、基于优先级的链排序和审计模式语义。
- **参考实现**：用于构建拦截器的多语言 SDK 库、示例拦截器（PII 编辑、模式验证、审计日志）、通用拦截器边车/代理运行时，以及用于拦截器调用和测试的 CLI 客户端。
- **跨领域问题**：传输级拦截点、基于网关的部署模式，以及与路由和策略层的交互（见相关小组）。
- **文档**：涵盖拦截器编写、部署模型（进程内、边车、远程服务）、链配置以及从临时中间件方法迁移的指南的规范部分。

### 范围外

- 客户端特定的钩子实现细节（例如，Claude Code 的内部钩子执行引擎）——WG 标准化协议级接口，而非主机内部实现。
- 传输层线路格式或会话模型更改（由传输工作组负责）。
- 超出 MCP 协议要求的通用中间件或代理基础设施。

### 相关小组

- **传输 WG** — 拦截器在 MCP 消息流上操作，其传递行为取决于传输；需要在传输级拦截点上进行协调。
- **网关 IG** — 网关是拦截器的关键部署模型；需要就基于网关的拦截器模式以及围绕路由、策略和可观察性的共享关注点进行协调。
