有效的调试在开发 MCP server 或将它们与应用程序集成时至关重要。本指南涵盖了 MCP 生态系统中可用的调试工具和方法。

## 调试工具概览

MCP 提供了多种不同级别的调试工具：

1. **[MCP Inspector](/docs/tools/inspector)**：交互式、传输无关的测试 UI。连接到 stdio 或 Streamable HTTP servers，调用 [tools](/specification/latest/server/tools)、[prompts](/specification/latest/server/prompts) 和 [resources](/specification/latest/server/resources)，并观察通知流。这应该是你的首选。
2. **Server 日志记录**：结构化日志输出到 stderr（stdio 传输）或通过 [`notifications/message`](/specification/latest/server/utilities/logging#log-message-notifications)（所有传输）。
3. **Client 开发者工具**：大多数 MCP clients 暴露日志和连接状态。下面以[在 Claude Desktop 中调试](#debugging-in-claude-desktop)为例，或查阅你的 client 文档。

## 实现日志记录

### 服务器端日志记录

当构建使用本地 [stdio 传输](/specification/latest/basic/transports#stdio) 的 server 时，所有记录到 stderr（标准错误）的消息将自动被 host 应用程序捕获。

<Warning>

本地 MCP server 不应将消息记录到 stdout（标准输出），因为这会干扰协议操作。

</Warning>

对于使用 [Streamable HTTP 传输](/specification/latest/basic/transports#streamable-http)的 servers，client 不会捕获 stderr。请使用下面的日志消息通知、你自己的服务器端日志聚合或标准 HTTP 工具（curl、浏览器 DevTools 网络面板）来检查请求、[`Mcp-Session-Id` 头](/specification/latest/basic/transports#session-management)和 SSE 流。

对于所有[传输](/specification/latest/basic/transports)，你还可以通过发送日志消息通知向 client 提供日志记录：

<CodeGroup>

```python Python
@server.tool()
async def my_tool(ctx: Context) -> str:
    await ctx.session.send_log_message(
        level="info",
        data="Server started successfully",
    )
    return "done"
```

```typescript TypeScript
await server.sendLoggingMessage({
  level: "info",
  data: "Server started successfully",
});
```

</CodeGroup>

MCP 定义了八个 [RFC 5424 严重级别](/specification/latest/server/utilities/logging#log-levels)（`debug` 到 `emergency`）。Clients 可以在运行时通过 [`logging/setLevel`](/specification/latest/server/utilities/logging#setting-log-level) 请求调整最低级别。

需要记录的重要事件：

- 初始化步骤
- 资源访问
- Tool 执行
- 错误条件
- 性能指标

## 常见问题

以下示例使用 Claude Desktop 的 [`claude_desktop_config.json`](/docs/develop/connect-local-servers)；相同原则适用于任何基于 stdio 的 MCP client。

### 工作目录

当 MCP client 启动 stdio server 时：

- 通过 client 配置启动的 servers 的工作目录可能未定义（如 macOS 上的 `/`），因为 client 可能从任何位置启动
- 始终在配置和 `.env` 文件中使用绝对路径以确保可靠操作
- 通过命令行直接测试 servers 时，工作目录将是你运行命令的位置

例如，在 `claude_desktop_config.json` 中，使用：

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/username/data"
      ]
    }
  }
}
```

而不是像 `./data` 这样的相对路径

### 环境变量

通过 stdio 启动的 MCP server 只自动继承有限的环境变量子集（确切的集合取决于平台）。

要覆盖默认变量或提供你自己的变量，你可以在 `claude_desktop_config.json` 中指定 `env` 键：

```json
{
  "mcpServers": {
    "myserver": {
      "command": "mcp-server-myapp",
      "env": {
        "MYAPP_API_KEY": "some_key"
      }
    }
  }
}
```

### Server 初始化

常见的初始化问题：

1. **路径问题**
   - Server 可执行文件路径不正确
   - 缺少必需文件
   - 权限问题
   - 尝试为 `command` 使用绝对路径

2. **配置错误**
   - JSON 语法无效
   - 缺少必填字段
   - 类型不匹配

3. **环境问题**
   - 缺少环境变量
   - 变量值不正确
   - 权限限制

### 连接问题

当 servers 无法连接时：

1. 检查 client 日志
2. 验证 server 进程正在运行
3. 使用 [Inspector](/docs/tools/inspector) 独立测试
4. 验证[协议兼容性](/specification/latest/basic/lifecycle#version-negotiation)
5. 检查[能力协商](/specification/latest/basic/lifecycle#capability-negotiation)：错误 [`-32602`](/specification/latest/basic/lifecycle#error-handling) 是标准 JSON-RPC "无效参数"代码，在许多上下文中都会返回。一个常见原因是 server 向未声明该能力的 client 发送 [sampling](/specification/latest/client/sampling) 或 [elicitation](/specification/latest/client/elicitation) 请求。检查 [`initialize` 交换](/specification/latest/basic/lifecycle#initialization)以验证双方声明了你期望的内容

## 在 Claude Desktop 中调试

Claude Desktop 是众多 MCP clients 之一。适用于 macOS 和 Windows。

### 检查 server 状态

点击聊天输入中的"添加文件、连接器等"加号图标，然后悬停在 **Connectors** 菜单上查看已连接的 servers 和可用 tools。

<img src="/images/available-mcp-tools.png" alt="可用的 MCP tools" />

### 查看日志

日志文件写入到：

- macOS：`~/Library/Logs/Claude`
- Windows：`%APPDATA%\Claude\logs`

<CodeGroup>

```bash macOS
tail -n 20 -F ~/Library/Logs/Claude/mcp*.log
```

```powershell Windows
type "$env:AppData\Claude\logs\mcp*.log"
```

</CodeGroup>

日志捕获：

- Server 连接事件
- 配置问题
- 运行时错误
- 消息交换

### 使用 Chrome DevTools

在 Claude Desktop 内访问 Chrome 的开发者工具以调查客户端错误：

1. 创建一个包含 `allowDevTools` 设置为 true 的 `developer_settings.json` 文件：

<CodeGroup>

```bash macOS
echo '{"allowDevTools": true}' > ~/Library/Application\ Support/Claude/developer_settings.json
```

```powershell Windows
'{"allowDevTools": true}' | Set-Content "$env:AppData\Claude\developer_settings.json"
```

</CodeGroup>

2. 打开 DevTools：`Command-Option-I`（macOS）或 `Ctrl+Alt+I`（Windows）

注意：你会看到两个 DevTools 窗口：

- 主内容窗口
- 应用标题栏窗口

使用 Console 面板检查客户端错误。

使用 Network 面板检查：

- 消息负载
- 连接时序

## 调试工作流

### 开发周期

1. 初始开发
   - 使用 [Inspector](/docs/tools/inspector) 进行基本测试
   - 实现核心功能
   - 添加日志记录点

2. 集成测试
   - 在你的目标 MCP client 中测试
   - 监控日志
   - 检查错误处理

### 测试变更

要高效测试变更：

- **配置变更**：重启 MCP client
- **Server 代码变更**：重启 client（对于 Claude Desktop，完全退出并重新打开；关闭窗口是不够的）
- **快速迭代**：开发期间使用 [Inspector](/docs/tools/inspector)

## 最佳实践

### 日志记录策略

1. **结构化日志**
   - 使用一致的格式
   - 包含上下文
   - 添加时间戳
   - 跟踪请求 ID

2. **错误处理**
   - 记录堆栈跟踪
   - 包含错误上下文
   - 跟踪错误模式
   - 监控恢复

3. **性能跟踪**
   - Log operation timing
   - Monitor resource usage
   - Track message sizes
   - Measure latency

### 安全考虑

调试时：

1. **敏感数据**
   - 清理日志
   - 保护凭证
   - 屏蔽个人信息

2. **访问控制**
   - 验证权限
   - 检查认证
   - 监控访问模式

有关 MCP 攻击向量和缓解措施的全面讨论，请参阅[安全最佳实践](/docs/tutorials/security/security_best_practices)。

## 获取帮助

遇到问题时：

1. **第一步**
   - 检查 server 日志
   - 使用 [Inspector](/docs/tools/inspector) 测试
   - 审查配置
   - 验证环境

2. **支持渠道**
   - [GitHub issues](https://github.com/modelcontextprotocol/modelcontextprotocol/issues)
   - [GitHub discussions](https://github.com/modelcontextprotocol/modelcontextprotocol/discussions)

3. **提供信息**
   - 日志摘录
   - 配置文件
   - 重现步骤
   - 环境详情
