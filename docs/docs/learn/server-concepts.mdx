MCP server 是通过标准化协议接口向 AI 应用程序暴露特定能力的程序。

常见示例包括用于文档访问的文件系统 server、用于数据查询的数据库 server、用于代码管理的 GitHub server、用于团队通信的 Slack server 以及用于日程安排的日历 server。

## 核心 Server 功能

Server 通过三个构建块提供功能：

| 功能          | 说明                                                                                                                      | 示例                                     | 控制者   |
| ------------- | ------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- | -------- |
| **Tools**     | 你的 LLM 可以主动调用的函数，并根据用户请求决定何时使用它们。Tools 可以写入数据库、调用外部 API、修改文件或触发其他逻辑。 | 搜索航班<br />发送消息<br />创建日历事件 | 模型     |
| **Resources** | 提供只读信息访问权限的被动数据源，用于提供上下文，如文件内容、数据库 schema 或 API 文档。                                 | 检索文档<br />访问知识库<br />读取日历   | 应用程序 |
| **Prompts**   | 预构建的指令模板，告诉模型如何使用特定的 tools 和 resources。                                                             | 规划假期<br />总结我的会议<br />起草邮件 | 用户     |

我们将使用一个假设场景来演示这些功能各自的作用，并展示它们如何协同工作。

### Tools

Tools 使 AI 模型能够执行操作。每个 tool 定义一个具有类型化输入和输出的特定操作。模型根据上下文请求 tool 执行。

#### Tools 的工作方式

Tools 是 LLM 可以调用的 schema 定义接口。MCP 使用 JSON Schema 进行验证。每个 tool 执行一个具有明确定义输入和输出的单一操作。Tools 在执行前可能需要用户同意，有助于确保用户对模型采取的操作保持控制。

**协议操作：**

| 方法         | 用途           | 返回值                       |
| ------------ | -------------- | ---------------------------- |
| `tools/list` | 发现可用 tools | 包含 schema 的 tool 定义数组 |
| `tools/call` | 执行特定 tool  | Tool 执行结果                |

**示例 tool 定义：**

```typescript
{
  name: "searchFlights",
  description: "Search for available flights",
  inputSchema: {
    type: "object",
    properties: {
      origin: { type: "string", description: "Departure city" },
      destination: { type: "string", description: "Arrival city" },
      date: { type: "string", format: "date", description: "Travel date" }
    },
    required: ["origin", "destination", "date"]
  }
}
```

#### 示例：旅行预订

Tools 使 AI 应用程序能够代表用户执行操作。在旅行规划场景中，AI 应用程序可能会使用多个 tools 来帮助预订假期：

**航班搜索**

```
searchFlights(origin: "NYC", destination: "Barcelona", date: "2024-06-15")
```

查询多家航空公司并返回结构化的航班选项。

**日历锁定**

```
createCalendarEvent(title: "Barcelona Trip", startDate: "2024-06-15", endDate: "2024-06-22")
```

在用户的日历中标记旅行日期。

**邮件通知**

```
sendEmail(to: "team@work.com", subject: "Out of Office", body: "...")
```

向同事发送自动的外出回复邮件。

#### 用户交互模型

Tools 是模型控制的，意味着 AI 模型可以自动发现和调用它们。然而，MCP 通过多种机制强调人工监督。

为了信任和安全，应用程序可以通过各种机制实现用户控制，例如：

- 在 UI 中显示可用 tools，使用户能够定义在特定交互中是否应提供某个 tool
- 单个 tool 执行时的批准对话框
- 预批准某些安全操作的权限设置
- 显示所有 tool 执行及其结果的活动日志

### Resources

Resources 提供对信息的结构化访问，AI 应用程序可以检索这些信息并将其作为上下文提供给模型。

#### Resources 的工作方式

Resources 从文件、API、数据库或 AI 需要理解上下文的任何其他来源暴露数据。应用程序可以直接访问这些信息并决定如何使用它们——无论是选择相关部分、使用 embedding 搜索，还是将其全部传递给模型。

每个 resource 都有一个唯一的 URI（例如 `file:///path/to/document.md`），并声明其 MIME 类型以进行适当的内容处理。

Resources 支持两种发现模式：

- **直接 Resources** - 指向特定数据的固定 URI。示例：`calendar://events/2024` - 返回 2024 年的日历可用性
- **Resource 模板** - 带参数的动态 URI，用于灵活查询。示例：
  - `travel://activities/{city}/{category}` - 按城市和类别返回活动
  - `travel://activities/barcelona/museums` - 返回巴塞罗那的所有博物馆

Resource 模板包含元数据，如标题、描述和预期的 MIME 类型，使其可发现且自文档化。

**协议操作：**

| 方法                       | 用途                   | 返回值                   |
| -------------------------- | ---------------------- | ------------------------ |
| `resources/list`           | 列出可用直接 resources | resource 描述符数组      |
| `resources/templates/list` | 发现 resource 模板     | resource 模板定义数组    |
| `resources/read`           | 检索 resource 内容     | 带元数据的 resource 数据 |
| `resources/subscribe`      | 监控 resource 变化     | 订阅确认                 |

#### 示例：获取旅行规划上下文

继续旅行规划示例，resources 为 AI 应用程序提供相关信息的访问：

- **日历数据**（`calendar://events/2024`）- 检查用户可用性
- **旅行文件**（`file:///Documents/Travel/passport.pdf`）- 访问重要文件
- **以往行程**（`trips://history/barcelona-2023`）- 参考过去的旅行和偏好

AI 应用程序检索这些 resources 并决定如何处理它们，无论是使用 embedding 或关键字搜索选择数据的子集，还是将原始数据直接传递给模型。

在这种情况下，它将日历数据、天气信息和旅行偏好提供给模型，使其能够检查可用性、查询天气模式并参考过去的旅行偏好。

**Resource 模板示例：**

```json
{
  "uriTemplate": "weather://forecast/{city}/{date}",
  "name": "weather-forecast",
  "title": "Weather Forecast",
  "description": "Get weather forecast for any city and date",
  "mimeType": "application/json"
}

{
  "uriTemplate": "travel://flights/{origin}/{destination}",
  "name": "flight-search",
  "title": "Flight Search",
  "description": "Search available flights between cities",
  "mimeType": "application/json"
}
```

这些模板支持灵活查询。对于天气数据，用户可以访问任何城市/日期组合的预报。对于航班，他们可以搜索任意两个机场之间的航线。当用户输入 "NYC" 作为 `origin` 机场并开始输入 "Bar" 作为 `destination` 机场时，系统可以建议 "Barcelona (BCN)" 或 "Barbados (BGI)"。

#### 参数补全

动态 resources 支持参数补全。例如：

- 为 `weather://forecast/{city}` 输入 "Par" 可能会建议 "Paris" 或 "Park City"
- 为 `flights://search/{airport}` 输入 "JFK" 可能会建议 "JFK - John F. Kennedy International"

该系统帮助发现有效值，无需了解精确格式。

#### 用户交互模型

Resources 是应用程序驱动的，赋予它们在如何检索、处理和呈现可用上下文方面的灵活性。常见的交互模式包括：

- 树形或列表视图，用于在熟悉的类似文件夹结构中浏览 resources
- 搜索和筛选界面，用于查找特定 resources
- 基于启发式或 AI 选择的自动上下文包含或智能建议
- 手动或批量选择界面，用于包含单个或多个 resources

应用程序可以自由地通过任何适合其需求的界面模式来实现 resource 发现。该协议不强制要求特定的 UI 模式，允许使用具有预览功能的 resource 选择器、基于当前对话上下文的智能建议、包含多个 resources 的批量选择，或与现有文件浏览器和数据浏览器的集成。

### Prompts

Prompts 提供可重用的模板。它们允许 MCP server 作者为某个领域提供参数化的 prompts，或展示如何最好地使用 MCP server。

#### Prompts 的工作方式

Prompts 是结构化的模板，定义预期的输入和交互模式。它们由用户控制，需要显式调用而不是自动触发。Prompts 可以感知上下文，引用可用的 resources 和 tools 来创建全面的工作流。类似于 resources，prompts 支持参数补全，以帮助用户发现有效的参数值。

**协议操作：**

| 方法           | 用途                 | 返回值                     |
| -------------- | -------------------- | -------------------------- |
| `prompts/list` | 发现可用 prompts     | prompt 描述符数组          |
| `prompts/get`  | 检索 prompt 详细信息 | 包含参数的完整 prompt 定义 |

#### 示例：简化工作流

Prompts 为常见任务提供结构化模板。在旅行规划上下文中：

**"规划假期" prompt：**

```json
{
  "name": "plan-vacation",
  "title": "Plan a vacation",
  "description": "Guide through vacation planning process",
  "arguments": [
    { "name": "destination", "type": "string", "required": true },
    { "name": "duration", "type": "number", "description": "days" },
    { "name": "budget", "type": "number", "required": false },
    { "name": "interests", "type": "array", "items": { "type": "string" } }
  ]
}
```

与非结构化的自然语言输入不同，prompt 系统支持：

1. 选择"规划假期"模板
2. 结构化输入：巴塞罗那，7 天，3000 美元，["海滩", "建筑", "美食"]
3. 基于模板的一致工作流执行

#### 用户交互模型

Prompts 由用户控制，需要显式调用。该协议赋予实现者自由，可以设计在应用程序中感觉自然的界面。关键原则包括：

- 轻松发现可用的 prompts
- 清晰描述每个 prompt 的功能
- 带有验证的自然参数输入
- 透明展示 prompt 的底层模板

应用程序通常通过各种 UI 模式暴露 prompts，例如：

- 斜杠命令（输入 "/" 查看可用 prompts，如 /plan-vacation）
- 命令面板用于可搜索的访问
- 针对常用 prompts 的专用 UI 按钮
- 建议相关 prompts 的上下文菜单

## 整合 Servers

MCP 的真正威力在于多个 servers 协同工作，通过统一界面组合它们的 specialized 能力。

### 示例：多 Server 旅行规划

考虑一个个性化的 AI 旅行规划器应用程序，包含三个连接的 servers：

- **Travel Server** - 处理航班、酒店和行程
- **Weather Server** - 提供气候数据和预报
- **Calendar/Email Server** - 管理日程和通信

#### 完整流程

1. **用户使用参数调用 prompt：**

   ```json
   {
     "prompt": "plan-vacation",
     "arguments": {
       "destination": "Barcelona",
       "departure_date": "2024-06-15",
       "return_date": "2024-06-22",
       "budget": 3000,
       "travelers": 2
     }
   }
   ```

2. **用户选择要包含的 resources：**
   - `calendar://my-calendar/June-2024`（来自 Calendar Server）
   - `travel://preferences/europe`（来自 Travel Server）
   - `travel://past-trips/Spain-2023`（来自 Travel Server）

3. **AI 使用 tools 处理请求：**

   AI 首先读取所有选定的 resources 以收集上下文——从日历中识别可用日期，从旅行偏好中了解偏好的航空公司和酒店类型，并从过去的行程中发现以前喜欢的景点。

   利用这些上下文，AI 然后执行一系列 Tools：
   - `searchFlights()` - 查询从纽约到巴塞罗那的航班
   - `checkWeather()` - 获取旅行日期的气候预报

   AI 然后使用这些信息创建预订和后续步骤，在必要时请求用户批准：
   - `bookHotel()` - 在指定预算内查找酒店
   - `createCalendarEvent()` - 将旅行添加到用户的日历中
   - `sendEmail()` - 发送包含旅行详情的确认邮件

**结果：** 通过多个 MCP servers，用户研究并预订了适合其日程的巴塞罗那之旅。"规划假期" prompt 引导 AI 将 Resources（日历可用性和旅行历史）与 Tools（搜索航班、预订酒店、更新日历）跨不同 servers 结合——收集上下文并执行预订。原本可能需要数小时的任务在几分钟内通过 MCP 完成。
