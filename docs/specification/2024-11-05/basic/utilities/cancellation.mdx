---
title: 取消
weight: 10
---

Model Context Protocol (MCP) 支持通过通知消息可选地取消正在进行的请求。任何一方都可以发送取消通知，表示应终止先前发出的请求。

## 取消流程

当一方想要取消正在进行的请求时，它发送一个 `notifications/cancelled` 通知，包含：

- 要取消的请求的 ID
- 可选的原因字符串，可以记录或显示

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/cancelled",
  "params": {
    "requestId": "123",
    "reason": "User requested cancellation"
  }
}
```

## 行为要求

1. 取消通知 **MUST** 仅引用以下请求：
   - 先前在同一方向上发出的
   - 被认为仍在进行中的
2. 客户端 **MUST NOT** 取消 `initialize` 请求
3. 取消通知的接收方 **SHOULD**：
   - 停止处理被取消的请求
   - 释放相关资源
   - 不发送被取消请求的响应
4. 接收方 **MAY** 忽略取消通知，如果：
   - 引用的请求未知
   - 处理已经完成
   - 请求无法取消
5. 取消通知的发送方 **SHOULD** 忽略之后到达的任何对该请求的响应

## 时序考虑

由于网络延迟，取消通知可能在请求处理完成后到达，并且可能在响应已发送之后到达。

双方 **MUST** 优雅地处理这些竞态条件：

```mermaid
sequenceDiagram
   participant Client
   participant Server

   Client->>Server: 请求 (ID: 123)
   Note over Server: 开始处理
   Client--)Server: notifications/cancelled (ID: 123)
   alt
      Note over Server: 处理可能在取消到达前已完成
   else 如果未完成
      Note over Server: 停止处理
   end
```

## 实现说明

- 双方 **SHOULD** 记录取消原因以便调试
- 应用程序 UI **SHOULD** 在请求取消时进行指示

## 错误处理

无效的取消通知 **SHOULD** 被忽略：

- 未知的请求 ID
- 已完成的请求
- 格式错误的通知

这保持了通知的"即发即弃"特性，同时允许异步通信中的竞态条件。
