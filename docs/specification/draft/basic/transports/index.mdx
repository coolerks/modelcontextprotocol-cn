本页定义了传输必须提供什么来承载 MCP 消息、标准传输绑定以及定义新传输的要求。

协议语义在每个传输上都是相同的。传输是一个**绑定**：它定义了消息如何帧封装和传递、请求元数据如何携带、以及取消和终止如何发出信号。它不定消息的含义：[消息模式](/specification/draft/basic/patterns)是核心协议的一部分，在每个绑定上都是相同的。绑定页面指定了标准传输：

1. [stdio](/specification/draft/basic/transports/stdio)：通过客户端启动的子进程的标准流进行换行符分隔的消息。
2. [Streamable HTTP](/specification/draft/basic/transports/streamable-http)：每个消息都是对单个 MCP 端点的 HTTP POST；回复以 JSON 对象或请求作用域的 SSE 流形式到达。

客户端和服务器也可以实现[自定义传输](#custom-transports)。

## 消息

MCP 使用 JSON-RPC 对消息进行编码。JSON-RPC 消息 **MUST** 使用 UTF-8 编码。

绑定 **MUST** 将客户端发送的*请求*和*通知*传递给服务器，并将服务器发送的*响应*和*通知*传递给客户端。没有其他消息方向存在：根据[消息模式](/specification/draft/basic/patterns)，服务器不发起 JSON-RPC 请求，客户端也不发送 JSON-RPC 响应。

## 请求元数据

所有协议元数据都在消息体中传输：每个请求在其 [`_meta.io.modelcontextprotocol/*`](/specification/draft/basic/index#meta) 字段中携带协议版本、客户端标识和客户端能力。

绑定 **MAY** 额外将选定的消息体字段镜像到信封元数据中。Streamable HTTP 传输将它们镜像到 [HTTP 头部](/specification/draft/basic/transports/streamable-http#request-metadata)中，以便中间件可以在不解析消息体的情况下路由和检查请求。消息体仍然是真实来源；镜像元数据的绑定定义了不匹配时的拒绝方式。

## 取消

每个绑定定义了客户端如何放弃正在进行的请求：在 stdio 上，客户端发送 `notifications/cancelled` 通知；在 Streamable HTTP 上，它关闭请求的响应流。协议级别的规则在所有地方都是相同的；请参见[取消](/specification/draft/basic/patterns/cancellation)。

## 自定义传输

客户端和服务器 **MAY** 实现额外的自定义传输机制以满足其特定需求。协议是传输无关的，可以在任何支持双向消息交换的通信通道上实现。

选择支持自定义传输的实现者 **MUST** 保留 JSON-RPC 消息格式、[消息模式](/specification/draft/basic/patterns)和按请求元数据模型。自定义传输 **SHOULD** 记录其连接建立、消息帧封装和取消模式，以促进互操作性。

在可靠的面向连接字节流（例如，Unix 域套接字或 TCP）上运行的自定义传输 **SHOULD** 重用 [stdio 帧封装](/specification/draft/basic/transports/stdio) 而不是定义新的：stdio 绑定只是在字节流上的换行符分隔的 JSON-RPC，只有其进程生命周期规则是标准流特有的。

## 向后兼容性

早期的协议修订版通过 `initialize` 握手建立连接范围的会话，并允许服务器发起 JSON-RPC 请求。与这些修订版互操作的客户端和服务器检测对方的时代并按照[版本控制：向后兼容性](/specification/draft/basic/versioning#backward-compatibility-with-initialization-based-versions)中的描述回退，其中包含一个供实现者参考的兼容性矩阵。每个绑定页面描述了其传输特定的检测机制。
