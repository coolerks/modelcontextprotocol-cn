`subscriptions/listen` 打开一个从服务器到客户端的长期存在的通知流。
与一次性请求不同，流保持开放并传递通知，直到客户端取消它。
它取代了之前的 `resources/subscribe` RPC 和 HTTP GET 端点。

## 打开流

客户端发送带有 `notifications` 过滤器的 `subscriptions/listen` 请求，
指定它希望接收哪些事件类型。服务器 **MUST NOT** 发送
客户端未明确请求的通知类型。

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "subscriptions/listen",
  "params": {
    "_meta": {
      "io.modelcontextprotocol/protocolVersion": "2026-07-28",
      "io.modelcontextprotocol/clientInfo": {
        "name": "ExampleClient",
        "version": "1.0.0"
      },
      "io.modelcontextprotocol/clientCapabilities": {}
    },
    "notifications": {
      "toolsListChanged": true,
      "resourceSubscriptions": ["file:///project/config.json"]
    }
  }
}
```

### 通知过滤器

| 字段                    | 类型       | 描述                                                    |
| ----------------------- | ---------- | ------------------------------------------------------- |
| `toolsListChanged`      | `boolean`  | 工具变更时接收 `notifications/tools/list_changed`       |
| `promptsListChanged`    | `boolean`  | 提示变更时接收 `notifications/prompts/list_changed`     |
| `resourcesListChanged`  | `boolean`  | 列表变更时接收 `notifications/resources/list_changed`   |
| `resourceSubscriptions` | `string[]` | 针对这些资源 URI 接收 `notifications/resources/updated` |

所有字段都是可选的。省略某个字段相当于不订阅该通知类型。

## 确认

服务器 **MUST** 发送 `notifications/subscriptions/acknowledged` 作为流上的第一条消息。
确认中的 `notifications` 字段反映服务器同意处理的子集 —
服务器不支持的通知类型被省略。

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/subscriptions/acknowledged",
  "params": {
    "_meta": {
      "io.modelcontextprotocol/subscriptionId": "1"
    },
    "notifications": {
      "toolsListChanged": true,
      "resourceSubscriptions": ["file:///project/config.json"]
    }
  }
}
```

客户端 **SHOULD** 检查已确认的过滤器与它请求的内容，并优雅处理任何不支持的类型。

## 接收通知

在流上传递的所有通知在 `_meta` 中都带有
`io.modelcontextprotocol/subscriptionId`，匹配打开该流的 `subscriptions/listen` 请求的 ID。
在 stdio 上，所有消息共享单个通道，客户端 **MUST** 使用此字段将通知与其原始订阅关联。

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/resources/updated",
  "params": {
    "_meta": {
      "io.modelcontextprotocol/subscriptionId": "1"
    },
    "uri": "file:///project/config.json"
  }
}
```

## 多个并发订阅

客户端 **MAY** 同时有多个活跃订阅 — 例如，
一个监听工具列表变更，另一个监听资源更新。
每个订阅由其 `subscriptions/listen` 请求的 JSON-RPC 请求 ID 标识，
流上的每个通知在 `io.modelcontextprotocol/subscriptionId` 中携带该 ID，
以便客户端可以解复用它们。

## 取消

订阅在以下情况结束：

- **客户端**取消它 — 关闭 SSE 流（HTTP）或发送引用 `subscriptions/listen` 请求 ID 的 `notifications/cancelled`（stdio）。
- **服务器**拆除它（例如，在关闭期间）— 它 **MUST** 关闭 SSE 流（HTTP）或发送引用 `subscriptions/listen` 请求 ID 的 `notifications/cancelled`（stdio）。
- 底层传输关闭（HTTP 超时、TCP 断开、stdio 进程退出）。

在 **stdio** 上，如果连接被终止然后重新建立，客户端 **MUST** 重新发送 `subscriptions/listen` 以重新建立其订阅 — 服务器在重新连接期间不保持任何订阅状态。

有关完整规则，请参见[取消][cancellation]。
