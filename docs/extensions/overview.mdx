# MCP 扩展

MCP 扩展是对规范的可选补充，定义了核心协议之外的能力。扩展可以实现模块化功能（例如身份验证等独立特性）、专业化功能（例如行业特定逻辑）或实验性功能（例如正在孵化以纳入核心的功能）。

扩展使用唯一的*扩展标识符*进行识别，格式为：`{供应商前缀}/{扩展名称}`，例如 `io.modelcontextprotocol/oauth-client-credentials`。标识符遵循与 [`_meta` 键](/specification/draft/basic/index#meta)相同的规则，带有强制性前缀。官方扩展使用 `io.modelcontextprotocol` 供应商前缀。

<Tip>

如果您正在构建第三方扩展，请使用您拥有的反向域名作为供应商前缀以避免冲突（类似于 Java 包命名）。例如，拥有 `example.com` 的公司应使用 `com.example/` 作为前缀（例如 `com.example/my-extension`）。

</Tip>

## 实验性扩展

实验性扩展为[工作组和兴趣组](/community/working-interest-groups)提供孵化途径，使其能够在正式 SEP 提交之前原型化想法并协作开发扩展概念。

实验性扩展仓库位于 MCP GitHub 组织下，以 `experimental-ext-` 为前缀（例如 `experimental-ext-interceptors`）。

### 基本规则

- 每个实验性扩展都需要关联一个工作组或兴趣组
- 仓库和发布的包需要明确标明其实验性状态（例如在 README 和包名称中）
- [核心维护者](/community/contributor-ladder#core-maintainer)保留对实验性扩展仓库的监督权，包括归档或删除的能力

### 晋升为官方状态

要将实验性扩展提升为官方状态，需要经过标准的 SEP 流程（扩展轨道）。您可以引用实验性仓库以及在孵化期间构建的任何参考实现来证明扩展的实用性。

## 创建扩展

官方扩展的生命周期遵循基于 SEP 的流程。详情请参阅 [SEP-2133: 扩展](/seps/2133-extensions)。

1. **提议**：使用[标准 SEP 指南](/community/sep-guidelines)在主 MCP 仓库中创建类型为**扩展轨道**的 SEP。
2. **实现**：在官方 SDK 中至少构建一个参考实现——这是在 SEP 可以被审查之前必需的。
3. **审查**：[核心维护者](/community/contributor-ladder#core-maintainer)审查 SEP 并拥有最终决定权。
4. **发布**：批准后，打开一个 PR 将扩展添加到扩展仓库中。
5. **采用**：之后，其他客户端、服务器和 SDK 也可以实现该扩展。

### 要求

- 扩展规范需要使用 RFC 2119 语言（MUST、SHOULD、MAY）
- 扩展必须关联一个工作组或兴趣组

### SDK 实现

SDK 可以选择实现扩展，但这对于协议合规性不是必需的。SDK 维护者对其支持的扩展拥有完全的自主权。如果 SDK 支持扩展，SDK 文档应列出支持的扩展。

<Note>

扩展默认始终处于禁用状态，需要开发者明确选择加入。

</Note>

### 演进

扩展独立于核心协议演进。更新由扩展仓库维护者管理，无需核心维护者审查。

尽管如此，向后兼容性很重要。当您需要更改扩展时，优先使用能力标志或扩展设置对象内的版本控制，而不是创建新的扩展标识符。如果无法避免破坏性变更，请使用新的标识符（例如 `io.modelcontextprotocol/my-extension-v2`）。

**破坏性变更**是指任何会导致现有实现失败或行为不正确的修改，包括：

- 删除或重命名字段
- 更改字段类型
- 改变现有行为的语义
- 添加新的必填字段

## 协商

客户端和服务器在[初始化握手](/specification/latest/basic/lifecycle)期间，通过各自能力中的 `extensions` 字段声明对扩展的支持。

### 客户端能力

客户端在 `initialize` 请求中声明扩展支持：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "roots": {
        "listChanged": true
      },
      "extensions": {
        "io.modelcontextprotocol/ui": {
          "mimeTypes": ["text/html;profile=mcp-app"]
        }
      }
    },
    "clientInfo": {
      "name": "ExampleClient",
      "version": "1.0.0"
    }
  }
}
```

### 服务器能力

服务器在 `initialize` 响应中声明扩展支持：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-06-18",
    "capabilities": {
      "tools": {},
      "extensions": {
        "io.modelcontextprotocol/ui": {}
      }
    },
    "serverInfo": {
      "name": "ExampleServer",
      "version": "1.0.0"
    }
  }
}
```

每个扩展指定其设置对象的模式；空对象表示无设置。

### 优雅降级

如果一方支持某个扩展而另一方不支持，支持方需要回退到核心协议行为，或者在扩展为强制性时以适当的错误拒绝请求。

良好的做法是在扩展中记录预期的回退行为。例如，提供 UI 增强工具的服务器应仍然为不支持 UI 扩展的客户端返回有意义的文本内容。另一方面，需要特定身份验证扩展的服务器可以拒绝来自不支持该扩展的客户端的连接。
